# Процесс команды relationship_diary

Поток одной фичи от идеи до деплоя. Каждый этап выполняет профильный субагент, читает артефакт предыдущего этапа и пишет свой.

```
  PO          BA           Архитектор     Лид         QA           Безопасник        DevOps
  идея ──▶ требования ──▶ спецификация ─▶ код ──▶ e2e-тесты ──▶ security-review ──▶ деплой
/intake   /analyze       /design        /build    /qa          /secure            /deploy
00-brief  01-requirements 02-spec       03-build  04-test-report 04b-security-review 05-deploy
                                      + frontend/ + backend/
        └────────────────────  /ship = весь конвейер автономно  ────────────────────┘
```

## Этапы
1. **INTAKE** (`/intake`) — оркестратор заводит папку фичи и бриф. Этапы не запускаются.
2. **ANALYZE** (`business-analyst`) — формализует требования и acceptance criteria.
3. **DESIGN** (`architect`) — техническая спецификация, контракты API/DynamoDB, план задач T-X, ADR.
4. **BUILD** (`dev-lead`) — реализация в `frontend/src/` и/или `backend/src/`, коммиты, build-notes.
5. **QA** (`qa-engineer`) — e2e/смоук против acceptance criteria, вердикт PASS/FAIL.
6. **SECURITY** (`security-engineer`) — security-review против модели угроз (IDOR, приватность полей, JWT, инвайты, push, утечки), вердикт PASS/FAIL. Обязательный гейт перед деплоем; можно гнать параллельно с QA.
7. **DEPLOY** (`devops`) — сборка фронта, SAM-деплой бэка, S3+CloudFront, SSM, наблюдаемость.

## Петли и возвраты
- **build ↔ qa:** QA FAIL → дефекты `[BUG]` возвращаются лиду → повтор QA. Максимум 3 круга, затем стоп и эскалация PO.
- **build ↔ security:** SECURITY FAIL → находки `[SEC]` возвращаются лиду → повтор security-review. Максимум 3 круга, затем стоп и эскалация PO.
- **design → analyze:** `[NEEDS-BA]` — требование нереализуемо, один возврат к BA.
- **build → design:** `[NEEDS-ARCH]` — расхождение контрактов, один возврат к архитектору.

## Когда будят PO (продукт-овнера)
Только `[BLOCKER]`, `[NEEDS-CEO-APPROVAL]` (боевой деплой в прод / изменение прод-данных / необратимое), исчерпанная петля дефектов, отсутствие критичных секретов. В остальных случаях команда решает сама и фиксирует решение в артефактах.

## Приватность (сквозное правило)
Приложение — для пары из двух человек. Личные поля (`free_thought`) партнёру не видны; общие (`note_to_partner`) — видны. Каждый этап обязан сохранять это разграничение.

## Состояние
Источник истины по этапам — `pipeline.md` в папке фичи. `/status` агрегирует все фичи в `docs/BACKLOG.md`.
