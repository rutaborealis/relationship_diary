# ADR 0002 — Инвалидация stateless-JWT после смены пароля через tokenVersion

- **Статус:** Accepted
- **Дата:** 2026-06-30
- **Контекст фичи:** `docs/features/password-reset/` (FR-14, AC-11)
- **Решение:** инженерное (архитектор/безопасник), PO-аппрув не требуется — необратимых/боевых изменений данных нет.

## Контекст

JWT сейчас stateless: `signToken` подписывает `{ userId, email }` с TTL 7 дней,
`requireAuth` лишь верифицирует подпись и issuer (`lib/jwt.ts`, `lib/auth-middleware.ts`).
Механизма отзыва нет. Требование FR-14/AC-11: после успешной смены пароля все
ранее выданные токены (на других устройствах) должны мгновенно перестать
действовать («logout на всех устройствах»). Со stateless-подписью без серверного
состояния это недостижимо — нужен серверный признак, проверяемый при каждом
аутентифицированном запросе.

## Решение

Ввести в профиль пользователя (`USER#<id>/PROFILE`) числовое поле **`tokenVersion`**
(default 1):

1. JWT-payload расширяется полем **`tv`** = текущий `tokenVersion` на момент выдачи
   (`signToken` в login и verify-email кладёт актуальное значение).
2. `requireAuth` после проверки подписи **дочитывает профиль** и сверяет
   `(payload.tv ?? 1) === (profile.tokenVersion ?? 1)`. Несовпадение → `401`.
3. `confirm-reset` инкрементит счётчик: `UpdateItem ... ADD tokenVersion :1` в той же
   операции, что обновляет `passwordHash`. Все прежние токены (с меньшим `tv`)
   немедленно становятся невалидными.
4. **Миграция не нужна:** отсутствие атрибута трактуется как `tv = 1` и при подписи,
   и при проверке. Существующие сессии продолжают работать до истечения 7 дней
   (их `tv` отсутствует == 1, профиль == 1).
5. Фронт уже выполняет авто-logout на `401` (`api/index.ts`) — отдельной обработки
   на клиенте AC-11 не требует.

## Последствия

**Плюсы:** закрывает FR-14/AC-11 точно и просто; одно целочисленное поле;
переиспользуемо для будущих «выйти со всех устройств» (смена пароля из настроек,
компрометация); без миграции данных; обратимо (снять проверку = вернуть stateless).

**Минусы (осознанные):**
- **+1 `GetItem` профиля на каждый аутентифицированный запрос.** Для приложения
  строго на 2 пользователей доп. чтения и стоимость ничтожны; большинство
  хендлеров и так читают профиль/данные из `DiaryMain` (можно позже схлопнуть
  чтение middleware и хендлера в одно). Латентность — единицы мс на on-demand-таблице.
- Частично подрывает «чистую statelessness» JWT — но это и есть цель (нужен отзыв).

## Альтернативы

- **`passwordChangedAt` (timestamp) vs `iat` токена.** Эквивалентно по эффекту
  (токен с `iat < passwordChangedAt` отвергается), тоже требует чтения профиля.
  Выбран `tokenVersion`: явный монотонный счётчик, не зависит от точности часов и
  гранулярности `iat` (секунды), и расширяем на «отозвать сейчас» без смены пароля.
- **Denylist отозванных токенов (DynamoDB + TTL).** Избыточно: хранить/чистить
  jti, та же стоимость чтения; для 2 юзеров оверинжиниринг.
- **Короткий TTL access + refresh-токены.** Большая переделка auth; не нужна для
  требуемой семантики и текущего масштаба.
- **Ничего не делать / полагаться на 7-дневный TTL.** Не закрывает FR-14
  (скомпрометированная сессия живёт до 7 дней). Отклонено.
