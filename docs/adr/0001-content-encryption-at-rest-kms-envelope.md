# ADR 0001 — Шифрование контента дневника at-rest через KMS envelope encryption

- **Статус:** Accepted
- **Дата:** 2026-06-30
- **Контекст фичи:** `docs/features/content-encryption/`
- **Решение PO:** at-rest server-side (E2E отклонён), зафиксировано 2026-06-30.

## Контекст

Контентные текстовые поля записей дня и «качеств» лежат в DynamoDB (`DiaryMain`)
в открытом виде. Угроза v1 — пассивное раскрытие данных «в покое»: дамп/бэкап
таблицы, доступ оператора, ошибочно открытый доступ. Нужно шифровать контент,
не ломая серверные фичи (доставка `note_to_partner` партнёру, календарь, push) и
не меняя клиент/контракты API. Бюджет минимальный (< $2/мес на пару).

E2E рассмотрен и **отклонён PO**: не подходит для PWA (мульти-устройство,
сброс пароля = безвозвратная потеря данных, push без текста, усложнённый шаринг
партнёру). Кандидат на отдельную будущую фичу.

## Решение

Envelope encryption на серверной стороне:

1. Один симметричный **KMS CMK** (`alias/diary-content-<stage>`, `SYMMETRIC_DEFAULT`,
   ротация ключа включена).
2. Lambda вызывает `kms:GenerateDataKey` → получает плейн-текст data key (DEK) и
   его зашифрованную версию (wrapped key). Контент шифруется **локально** через
   `AES-256-GCM` (Node `crypto`). KMS на каждое поле не вызывается.
3. **Кэш data key в памяти тёплого контейнера** (TTL, лимит использований) для
   шифрования; **кэш расшифрованных DEK по их wrapped-байтам** для чтения. KMS-вызовы
   схлопываются до единиц в день на пару → стоимость символическая (доминирует
   $1/мес за CMK).
4. Каждое зашифрованное поле хранится как DynamoDB **Map** с маркером версии:
   `{ enc_v: 1, iv, tag, ct, dk }` (base64). Маркер `enc_v` различает форматы
   (FR-8) и поддерживает будущую ротацию.
5. **Детекция legacy:** значение-строка = plaintext (читается как есть, FR-9);
   значение-Map с `enc_v` = ciphertext. Ленивая миграция on-write (FR-10) +
   идемпотентный разовый backfill (FR-11).
6. **Контракты API не меняются** — клиент шлёт и получает plaintext; шифрование
   живёт только в слое хранения.

## Последствия

**Плюсы:** закрывает заявленную угрозу; сохраняет все серверные фичи и приватность
полей (`free_thought` не расшифровывается для партнёра); дёшево и прозрачно для
PWA; простое восстановление (ключ у сервиса); версионирование под ротацию.

**Минусы (осознанные):** сервер технически может расшифровать контент — от
инсайдера с одновременным доступом к рантайму Lambda и KMS не защищает (это вне
модели угроз v1, домен E2E). Дублирование wrapped-DEK в каждом поле (~+200 байт/поле)
— приемлемо при текущем объёме; при необходимости оптимизируется до item-level DEK.

## Альтернативы

- **KMS Decrypt на каждое поле каждого чтения** — отклонено: латентность и стоимость.
- **DynamoDB-native encryption at rest (AWS-owned/managed key)** — уже есть по
  умолчанию, но не закрывает «оператор с правами на таблицу / дамп через API
  читает plaintext»: значения возвращаются расшифрованными любому с доступом к
  таблице. Нужен прикладной слой шифрования. Отклонено как недостаточное.
- **E2E (ключ у пары)** — отклонено PO (см. контекст).
