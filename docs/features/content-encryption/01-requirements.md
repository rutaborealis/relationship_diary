# Требования

> Автор: business-analyst · Вход: 00-brief.md

## 1. Проблема и контекст

Дневник пары хранит крайне чувствительный личный контент: настроение, благодарности, заметки о близости, заметки партнёру и строго личные мысли (`free_thought`). Сегодня все эти поля лежат в DynamoDB (`DiaryMain`) **в открытом виде**. Любой, кто получит доступ к таблице — оператор с правами на БД, утечка дампа/бэкапа, скомпрометированные IAM-креды, ошибочно открытый доступ — читает интимный контент пары как обычный текст.

Это прямой риск приватности и доверия к продукту, чья единственная ценность — безопасное приватное пространство для двоих. Нужно ввести шифрование пользовательского контента так, чтобы значение в хранилище нельзя было прочитать как открытый текст, **не сломав** текущую модель приватности (личное vs общее) и существующие серверные фичи (доставка партнёру, push, календарь).

Контекст продукта: приложение строго на двух пользователей (пара), serverless-стек (Lambda + DynamoDB), бюджет минимальный.

## 2. Цель и метрики успеха

**Цель:** чувствительный текстовый контент дневника не хранится и не бэкапится в открытом виде; при этом UX и приватность полей не меняются для пользователей.

Метрики успеха:
- **100%** новых и мигрированных записей: контентные текстовые поля в DynamoDB — ciphertext (выборочная ручная/скриптовая проверка значений в таблице не даёт читаемого текста).
- **0** регрессий приватности: партнёр по-прежнему видит `note_to_partner`, никогда не видит `free_thought` (покрыто e2e).
- **0** видимой деградации UX: время сохранения и открытия записи на мобильном не вырастает заметно для пользователя (ориентир: дополнительная серверная задержка на шифрование/дешифрование < 150 мс на запись).
- Прирост стоимости AWS от шифрования — **символический** (ориентир < $2/мес на пару при выбранной at-rest модели).

## 3. Scope

**Входит:**
- Шифрование контентных текстовых полей записи дня (`ENTRY#<date>`): `mood_text`, `noticed_1..5`, `gratitude_1..5`, `closeness_text`, `note_to_partner`, `free_thought`.
- Шифрование текста пользовательских «качеств» (`QUALITY#<id>.text`) — это тоже личный контент.
- Прозрачное расшифрование на серверной стороне при чтении (`entries/get`, `qualities`), с сохранением текущих правил видимости полей.
- Стратегия миграции уже существующих plaintext-записей.
- Маркировка версии шифрования на каждом зашифрованном элементе для совместимости и будущей ротации.

**НЕ входит (out of scope):**
- Шифрование служебных метаданных и ключей доступа: `PK`, `SK`, `date`, `user_id`, `updated_at`, `saved_at`, `mood_level`, `gratitude_said`, `shared`, GSI-поля. Они нужны серверу для индексации, выборок, календаря и логики и **остаются в открытом виде**.
- Шифрование PII-полей профиля (`email`, `name`, `gender`; `passwordHash` уже хеш) — отдельная задача, не контент дневника.
- Полнотекстовый поиск по зашифрованному контенту (сейчас поиска по контенту в продукте нет).
- Передача текста записи внутри push-уведомления (сейчас push **не содержит** контент записи — только факт «партнёр заполнил дневник»; это сохраняется).
- Аппаратные ключи, парольная E2E-схема с восстановлением через recovery-код — кандидаты на отдельную фичу, см. открытый вопрос про E2E.
- Ротация мастер-ключа как регулярный процесс (закладываем версионирование, но сам прогон ротации — вне scope первой поставки).

## 4. Пользовательские сценарии

- Как **пользователь**, я хочу, чтобы мои личные мысли (`free_thought`) хранились в зашифрованном виде, чтобы даже при утечке БД их нельзя было прочитать.
- Как **пользователь**, я хочу, чтобы запись сохранялась и открывалась так же быстро и без изменений в интерфейсе, чтобы шифрование было для меня незаметным.
- Как **партнёр**, я хочу по-прежнему видеть `note_to_partner` второй половины, чтобы шифрование не лишало нас общих заметок.
- Как **пользователь**, я хочу быть уверен, что партнёр не получит доступ к моему `free_thought`, чтобы личное оставалось личным даже после внедрения шифрования.
- Как **владелец продукта**, я хочу, чтобы старые записи, созданные до шифрования, тоже стали защищены, чтобы не осталось «дыр» с открытым текстом.

## 5. Функциональные требования

- **FR-1.** Перед записью в DynamoDB сервер обязан шифровать значения всех контентных текстовых полей записи дня: `mood_text`, `noticed_1`, `noticed_2`, `noticed_3`, `noticed_4`, `noticed_5`, `gratitude_1`, `gratitude_2`, `gratitude_3`, `gratitude_4`, `gratitude_5`, `closeness_text`, `note_to_partner`, `free_thought`.
- **FR-2.** Сервер обязан шифровать поле `text` сущности `QUALITY#<id>` тем же механизмом.
- **FR-3.** Метаданные и ключевые поля НЕ шифруются: `PK`, `SK`, `date`, `user_id`, `updated_at`, `saved_at`, `mood_level`, `gratitude_said`, `shared`, любые поля, участвующие в GSI/выборках.
- **FR-4.** Пустые/`null`/отсутствующие контентные поля не шифруются и сохраняют семантику «значения нет» (пустая строка/`null` не должна превращаться в непустой ciphertext).
- **FR-5.** При чтении собственной записи (`entries/get`, владелец) сервер расшифровывает все контентные поля и возвращает их в открытом виде в JSON-ответе, как сегодня.
- **FR-6.** При чтении записи партнёра сервер расшифровывает только разрешённые к показу поля и возвращает `note_to_partner` партнёру в открытом виде. Поле `free_thought` партнёру **не возвращается вовсе** (как сейчас — `undefined`), независимо от шифрования.
- **FR-7.** Правило черновика сохраняется: если `shared === false`, запись партнёра не отдаётся целиком (как сегодня). Шифрование не меняет эту логику, т.к. `shared` — незашифрованный флаг.
- **FR-8.** Каждый зашифрованный элемент помечается версией схемы шифрования (например, поле-маркер `enc_v`), чтобы сервер мог различать форматы и поддерживать миграцию/ротацию.
- **FR-9.** При чтении элемента без маркера шифрования (legacy plaintext) сервер обязан корректно отдать его контент как есть (обратная совместимость на период миграции).
- **FR-10.** При следующем сохранении legacy-записи её контентные поля должны быть зашифрованы (ленивая миграция «on write»).
- **FR-11.** Должен существовать разовый миграционный прогон (скрипт/одноразовая Lambda), который перешифровывает все существующие plaintext-записи и качества обеих учёток пары в зашифрованный формат и проставляет маркер версии. Прогон идемпотентен (повторный запуск не портит уже зашифрованные элементы).
- **FR-12.** Дешифрование должно быть устойчивым к ошибке: при невозможности расшифровать конкретное поле сервер логирует инцидент (без утечки контента в логи) и возвращает контролируемую ошибку/пустое значение, не роняя весь ответ другими полями.
- **FR-13.** Контент в открытом виде не должен попадать в логи приложения, в тела ошибок и в push-payload.
- **FR-14.** Календарь (`entries/calendar`) и индикаторы заполненности продолжают работать на незашифрованных метаданных (наличие записи, `shared`), не требуя дешифрования контента.

## 6. Нефункциональные требования

- **Приватность полей (личное vs общее с партнёром):** разграничение неизменно. `free_thought` — личное, не покидает контур автора и не отдаётся партнёрскому ответу. `note_to_partner` — общее, расшифровывается и отдаётся партнёру. Выбранная модель шифрования обязана технически поддерживать оба сценария (at-rest server-side поддерживает «из коробки»; E2E потребует раздельных ключей для shared/private полей — см. §8).
- **Модель угроз (целевая для v1):** защита от пассивного раскрытия данных «в покое» — компрометация дампа/бэкапа DynamoDB, доступ оператора к таблице, ошибочно открытый доступ. НЕ в целях v1: защита от одновременной компрометации рантайма Lambda и ключа KMS (это уже про E2E, см. §8).
- **PWA/мобильный UX:** изменения шифрования прозрачны для клиента — формат API-ответов и форм не меняется; на клиенте никакой криптографии и управления ключами в v1 (при выбранной at-rest модели). Сохранение/открытие записи без заметной для пользователя задержки.
- **Доставка push / офлайн:** push не несёт контент записи и остаётся без изменений. Офлайн-черновики на клиенте работают как раньше (клиент шлёт открытый текст по HTTPS, шифрует сервер). HTTPS-канал (CloudFront/API GW) обязателен — шифрование в хранилище не заменяет шифрование в транзите.
- **Доступность только двум пользователям:** доступ к расшифрованному контенту авторизуется через JWT и `partnerId`; шифрование — дополнительный слой, не замена авторизации (IDOR-проверки остаются).
- **Производительность:** дополнительная серверная задержка на (де)шифрование записи — ориентир < 150 мс; не должно требовать синхронного внешнего вызова на каждое поле каждого чтения, если это бьёт по latency (envelope-шифрование с кэшированием data key — на усмотрение архитектора).
- **Стоимость AWS:** решение должно укладываться в минимальный бюджет (ориентир < $2/мес на пару). KMS-вариант (1 CMK + envelope encryption) предпочтителен по цене; вариант «KMS Decrypt на каждое поле каждого чтения» нежелателен из-за стоимости/латентности.
- **Восстановление доступа:** в at-rest модели потеря пользователем пароля/устройства НЕ ведёт к потере данных (ключ у сервиса). Это требование явно влияет на выбор модели — см. §8.
- **Аудит/проверяемость:** безопасник должен иметь возможность убедиться в шифровании, прочитав «сырое» значение поля в DynamoDB и не получив открытый текст.

## 7. Acceptance criteria

- **AC-1.** Given пользователь сохранил запись с заполненным `free_thought`, When безопасник читает соответствующий item напрямую в DynamoDB, Then значение `free_thought` не читается как открытый текст и присутствует маркер версии шифрования `enc_v`. *(FR-1, FR-8)*
- **AC-2.** Given у пользователя есть зашифрованная запись, When пользователь открывает свою запись через `entries/get`, Then все контентные поля (включая `free_thought`) возвращаются в открытом виде, как до шифрования. *(FR-5)*
- **AC-3.** Given партнёр заполнил `note_to_partner` в общей (`shared !== false`) записи, When второй пользователь открывает день партнёра, Then он видит расшифрованный текст `note_to_partner`. *(FR-6)*
- **AC-4.** Given партнёр заполнил `free_thought`, When второй пользователь запрашивает запись партнёра, Then поле `free_thought` отсутствует в ответе (не возвращается ни в каком виде). *(FR-6)*
- **AC-5.** Given запись партнёра имеет `shared === false`, When второй пользователь запрашивает её, Then запись не отдаётся целиком (как сегодня), независимо от шифрования. *(FR-7)*
- **AC-6.** Given существует legacy-запись в открытом виде без маркера шифрования, When её открывает владелец, Then контент корректно отображается (обратная совместимость). *(FR-9)*
- **AC-7.** Given legacy plaintext-запись, When владелец её пересохраняет, Then в DynamoDB её контентные поля становятся зашифрованными и получают маркер `enc_v`. *(FR-10)*
- **AC-8.** Given в таблице есть и plaintext, и уже зашифрованные записи, When выполняется миграционный прогон, Then все контентные поля становятся зашифрованными; повторный запуск прогона не повреждает уже зашифрованные данные. *(FR-11)*
- **AC-9.** Given поле `mood_text` пустое/`null`, When запись сохраняется, Then поле не превращается в непустой ciphertext и при чтении остаётся пустым/`null`. *(FR-4)*
- **AC-10.** Given сохранена зашифрованная запись, When проверяются метаданные item, Then `mood_level`, `shared`, `gratitude_said`, `date`, `saved_at`, `updated_at` остаются в открытом виде и пригодны для выборок/календаря. *(FR-3, FR-14)*
- **AC-11.** Given пользователь сохраняет запись и срабатывает push партнёру, When инспектируется payload уведомления, Then он не содержит контента записи (только факт заполнения). *(FR-13)*
- **AC-12.** Given при чтении одно поле невозможно расшифровать, When формируется ответ, Then остальные поля возвращаются корректно, инцидент логируется без раскрытия контента, ответ не падает целиком. *(FR-12)*

## 8. Допущения и открытые вопросы

**Принятые допущения:**
- A1. Шифруется только текстовый пользовательский контент (см. FR-1, FR-2). Числовые/булевы/служебные поля и ключи — нет (FR-3), т.к. они нужны для индексации, выборок и календаря.
- A2. Push-уведомления не несут текст записи и остаются как есть; «push с текстом записи» из брифа — гипотетический сценарий, в текущем продукте его нет, поэтому он не блокируется ни одной из моделей.
- A3. Объёмы данных малы (пара, по одной записи в день) → миграция и (де)шифрование не создают нагрузки; допустима ленивая миграция «on write» + разовый прогон скриптом.
- A4. Клиент в v1 не участвует в криптографии и управлении ключами (следствие рекомендованной at-rest модели). Формат API не меняется.
- A5. Маркер версии шифрования (`enc_v`) добавляется на элемент для совместимости и будущей ротации ключа; конкретный алгоритм/формат (envelope, AES-GCM, per-item data key и т.п.) — зона ответственности архитектора, не требований.
- A6. Канал в транзите уже защищён HTTPS (CloudFront/API GW); шифрование at-rest его дополняет, а не заменяет.
- A7. Поле `text` сущности `QUALITY` считается чувствительным контентом и включено в scope наравне с полями записи.
