# Требования

> Автор: business-analyst · Вход: 00-brief.md

## 1. Проблема и контекст

Сейчас в приложении нет механизма восстановления пароля. Аутентификация — это `email` + `password` (хранится `passwordHash`, bcrypt, в `USER#<id>/PROFILE`). Если пользователь забыл пароль, войти невозможно и сбросить его нечем — единственный путь к данным потерян.

Дневник пары ведут годами; потеря доступа к аккаунту = потеря всего личного контента и разрыв «общего пространства» пары (партнёр продолжает писать, а вторая половина выпадает). Это базовая ожидаемая функция аутентификации, и её отсутствие — реальная боль и прямая причина потери аккаунтов.

Контекст: приложение строго на двух пользователей; уже есть отлаженный канal email-кодов — регистрация шлёт 6-значный код через Resend (`VERIFY#<email>/CODE`, TTL 15 мин, `lib/ses.ts`). Логично переиспользовать тот же канал и тот же UX для сброса пароля. Поскольку это фича аутентификации, безопасность — первоклассное требование (защита от энумерации аккаунтов, от перебора кода, одноразовость, инвалидация сессий).

## 2. Цель и метрики успеха

**Цель:** пользователь, забывший пароль, может самостоятельно (self-service) восстановить доступ к аккаунту по email, не теряя данные дневника и не создавая новых рисков безопасности.

Метрики успеха:
- **Восстанавливаемость:** доля начатых сбросов, завершившихся успешным входом с новым паролем — **≥ 80%** (остальное — брошенные/истёкшие коды).
- **Снижение «мёртвых» аккаунтов:** **0** обращений «забыл пароль, не могу войти» без технического пути решения (раньше — тупик).
- **Безопасность (отрицательные метрики):** **0** способов через ответ API определить, зарегистрирован ли email; **0** успешных переборов кода в пределах лимита попыток (подтверждается security-review).
- **Без регрессий:** существующие флоу login / register / verify-email и хранение `passwordHash` работают как прежде (покрыто e2e/смоук).

## 3. Scope

**Входит:**
- Экран/шаг «Забыли пароль?» — точка входа со страницы логина (`LoginPage`).
- Запрос сброса по email: пользователь вводит email → бэкенд (если аккаунт существует и подтверждён) отправляет письмо с одноразовым кодом сброса через Resend.
- Единый, не раскрывающий существование аккаунта ответ на запрос сброса (anti-enumeration).
- Шаг ввода кода + установки нового пароля: проверка кода (одноразового, с TTL, с лимитом попыток) и сохранение нового `passwordHash`.
- Требования к новому паролю (согласованы с регистрацией) и подтверждение пароля на клиенте.
- Инвалидация ранее выданных сессий/JWT пользователя после успешной смены пароля (требование; механизм — к архитектору, см. §8).
- Защита от перебора и злоупотреблений: лимит попыток ввода кода + ограничение частоты запросов сброса (rate limiting).
- Письма на русском, в стиле существующего письма верификации.
- Аудит/логирование событий сброса без утечки секретов.

**НЕ входит (out of scope):**
- Смена пароля залогиненным пользователем из настроек (knowledge-based change при известном старом пароле) — отдельная фича.
- Двухфакторная аутентификация (2FA / TOTP / passkeys) и «магические ссылки» как замена паролю.
- Восстановление по SMS / резервным кодам / вопросам безопасности.
- Смена email, объединение/перенос аккаунтов, удаление аккаунта.
- Уведомление **партнёра** о том, что вторая половина сбросила пароль (партнёр в флоу сброса не участвует; см. допущение A7).
- Капча / интеграция внешнего anti-bot сервиса (рассматривается как опция архитектором, но не обязательна для v1; базовый rate limiting обязателен).

## 4. Пользовательские сценарии

- Как **пользователь, забывший пароль**, я хочу запросить восстановление по своему email, чтобы вернуть доступ к дневнику.
- Как **пользователь**, я хочу получить код сброса на ту же почту, что и при регистрации, чтобы процесс был знакомым и быстрым.
- Как **пользователь**, я хочу задать новый пароль и сразу войти, чтобы не проходить отдельный логин после сброса (или с понятным переходом ко входу).
- Как **пользователь, заботящийся о безопасности**, я хочу, чтобы после смены пароля все ранее открытые сессии (на других устройствах) перестали действовать, чтобы выгнать того, кто мог получить доступ.
- Как **владелец продукта**, я хочу, чтобы по ответу системы нельзя было узнать, зарегистрирован ли email, чтобы не давать атакующему список существующих аккаунтов.
- Как **владелец продукта**, я хочу, чтобы код сброса нельзя было подобрать перебором, чтобы чужой аккаунт нельзя было захватить.

## 5. Функциональные требования

**Запрос сброса**
- **FR-1.** Должен существовать публичный (без аутентификации) эндпоинт запроса сброса пароля, принимающий `email`. Email нормализуется так же, как везде (`trim().toLowerCase()`).
- **FR-2.** Ответ на запрос сброса **не должен раскрывать**, существует ли аккаунт: для существующего, несуществующего и неподтверждённого email возвращается **одинаковый** успешный ответ и одинаковое время реакции (без таймингового различия), с нейтральным сообщением вида «Если такой email зарегистрирован, мы отправили инструкции».
- **FR-3.** Письмо с кодом сброса отправляется **только** если email принадлежит существующему пользователю с `emailVerified === true`. Для несуществующего/неподтверждённого email письмо не отправляется (но ответ API тот же — см. FR-2).
- **FR-4.** Код сброса генерируется криптостойким источником случайности (а не `Math.random`), одноразовый, с коротким TTL. По умолчанию: 6-значный числовой код, TTL **15 минут** (согласовано с верификацией почты). *(Длина/формат и переход на «ссылку с токеном» вместо кода — на усмотрение архитектора; требование — криптостойкость, одноразовость, короткий TTL.)*
- **FR-5.** Активным считается не более одного кода сброса на email одновременно: новый запрос инвалидирует/перезаписывает предыдущий невыполненный код (запрос «отправить снова» = новый код, старый перестаёт действовать).
- **FR-6.** Частота запросов сброса на один email и/или с одного клиента должна быть ограничена (rate limiting), чтобы предотвратить email-флуд и злоупотребление каналом Resend. Конкретные пороги — к архитектору; требование — лимит существует и срабатывает.

**Установка нового пароля**
- **FR-7.** Должен существовать публичный эндпоинт подтверждения сброса, принимающий `email`, `code` (или токен) и `newPassword`.
- **FR-8.** Код валиден только если: совпадает с выданным, не истёк по TTL, не был уже использован, и не превышен лимит попыток ввода. Сравнение кода должно быть устойчиво к таймингу.
- **FR-9.** Должен действовать лимит попыток ввода кода (по умолчанию: **5** неверных попыток на код), после исчерпания которого код инвалидируется и требуется запросить новый. Сообщения об ошибке кода не должны раскрывать, существует ли аккаунт сверх того, что уже известно из факта получения письма.
- **FR-10.** Новый пароль должен удовлетворять политике паролей, согласованной с регистрацией: минимум **8 символов** (как в `register.ts`). Слишком короткий/пустой пароль отклоняется с понятной ошибкой.
- **FR-11.** Клиент должен запрашивать подтверждение пароля (ввод дважды) и не отправлять запрос при несовпадении; сервер не полагается на клиентскую проверку и валидирует политику самостоятельно.
- **FR-12.** При успешной валидации сервер пересчитывает `passwordHash` (bcrypt, те же `saltRounds`, что в конфиге) и атомарно обновляет `USER#<id>/PROFILE`. Прежний `passwordHash` нигде не сохраняется в открытом виде.
- **FR-13.** После успешной смены пароля код сброса немедленно становится недействительным (удаляется/помечается использованным); повторное использование того же кода невозможно.
- **FR-14.** После успешной смены пароля **все ранее выданные сессии/JWT пользователя должны перестать действовать** (logout на всех устройствах). JWT сейчас stateless без механизма отзыва — конкретная реализация инвалидации к архитектору/безопаснику (см. §8).
- **FR-15.** Пользователь, не имеющий действующей сессии, должен иметь возможность войти по новому паролю сразу после сброса (либо сервер выдаёт новый токен по завершении сброса, либо происходит редирект на логин — на усмотрение UX/архитектора).

**Уведомления и аудит**
- **FR-16.** События запроса сброса, успешной и неуспешной смены пароля, исчерпания лимита попыток должны логироваться для аудита **без записи секретов** (в логи не попадают: сам код/токен, новый пароль, `passwordHash`).
- **FR-17.** (Рекомендуется) После успешной смены пароля пользователю отправляется информационное письмо «ваш пароль был изменён» как сигнал на случай несанкционированного сброса. Помечено как желательное; приоритет — к PO/архитектору, не блокирует v1.

**UX / точки входа**
- **FR-18.** На странице логина должна быть видимая ссылка «Забыли пароль?», ведущая в флоу сброса.
- **FR-19.** Флоу на клиенте (PWA): экран ввода email → экран ввода кода + нового пароля (+ подтверждение) → завершение (вход или переход к логину). UX и визуальный стиль согласованы с `VerifyEmailPage`/`LoginPage`.
- **FR-20.** Если в письме используется ссылка с токеном (вариант архитектора), открытие ссылки в PWA/браузере должно корректно вести пользователя на экран установки нового пароля (по аналогии с обработкой invite-токена в `main.tsx`/`App.tsx`).

## 6. Нефункциональные требования

- **Приватность полей (личное vs общее с партнёром):** фича не читает и не показывает контент дневника; новых полей записи не вводит. Разграничение `free_thought` (личное) / `note_to_partner` (общее) не затрагивается. Партнёр **не должен** получать никакой информации о том, что вторая половина сбрасывала пароль (см. out-of-scope, A7).
- **Безопасность (первоклассно):**
  - Anti-enumeration: одинаковый ответ и тайминг для существующего/несуществующего email (FR-2).
  - Код криптостойкий, одноразовый, короткий TTL (FR-4), сравнение устойчиво к таймингу (FR-8).
  - Защита от перебора: лимит попыток на код (FR-9) + rate limiting на запросы (FR-6).
  - Инвалидация активных сессий после смены пароля (FR-14).
  - Секреты не попадают в логи/тела ошибок (FR-16); канал — только HTTPS (CloudFront/API GW).
- **PWA/мобильный UX:** флоу из 1–2 коротких экранов, удобный ввод 6-значного кода с цифровой клавиатурой (`inputMode="numeric"`), понятные ошибки и состояние загрузки; работает в standalone-PWA на телефоне.
- **Доставка email / офлайн:** письмо идёт через Resend (как verify-email). Возможна задержка/попадание в спам — UX даёт «Отправить снова» (с учётом FR-5/FR-6). Запрос сброса требует онлайна; офлайн-обработка не нужна.
- **Доступность только двум пользователям:** сброс не должен открывать обходной путь к чужому аккаунту/партнёрским данным; авторизация и разграничение пары остаются в силе.
- **Производительность и стоимость AWS:** новые эндпоинты — лёгкие Lambda-вызовы; reuse существующих DynamoDB-таблиц и Resend (несколько писем в год на пользователя) — стоимость практически нулевая. Anti-timing-задержка не должна делать запрос ощутимо медленным для легитимного пользователя.

## 7. Acceptance criteria

- **AC-1.** Given email **зарегистрирован и подтверждён**, When пользователь запрашивает сброс, Then он получает письмо с одноразовым кодом, а API возвращает нейтральный успешный ответ. *(FR-1, FR-3, FR-4)*
- **AC-2.** Given email **не зарегистрирован**, When отправляется запрос сброса, Then API возвращает **точно такой же** успешный ответ, как для существующего email, и письмо не отправляется. *(FR-2, FR-3)*
- **AC-3.** Given email зарегистрирован, но `emailVerified === false`, When отправляется запрос сброса, Then ответ такой же нейтральный и письмо сброса не отправляется. *(FR-3)*
- **AC-4.** Given пользователь получил код сброса, When он вводит верный код и валидный новый пароль, Then `passwordHash` обновляется и пользователь может войти по новому паролю. *(FR-7, FR-10, FR-12, FR-15)*
- **AC-5.** Given пользователь сменил пароль, When он пытается войти со **старым** паролем, Then вход отклоняется. *(FR-12)*
- **AC-6.** Given код сброса был успешно использован, When тот же код вводится повторно, Then он отклоняется как недействительный. *(FR-13)*
- **AC-7.** Given код сброса выдан, When прошло больше TTL (15 мин), Then код отклоняется как истёкший и требуется запросить новый. *(FR-4, FR-8)*
- **AC-8.** Given пользователь вводит неверный код, When число неверных попыток достигает лимита (5), Then код инвалидируется и дальнейшие попытки по нему отклоняются до запроса нового кода. *(FR-9)*
- **AC-9.** Given пользователь повторно запрашивает код («отправить снова»), When выдаётся новый код, Then предыдущий код перестаёт действовать. *(FR-5)*
- **AC-10.** Given на email/клиент уже отправлено много запросов сброса за короткое время, When отправляется ещё один, Then срабатывает rate limiting (запрос ограничивается без раскрытия существования аккаунта). *(FR-6)*
- **AC-11.** Given у пользователя есть активная сессия/JWT на другом устройстве, When он успешно сбрасывает пароль, Then прежняя сессия/JWT перестаёт работать (запросы с ней получают 401). *(FR-14)*
- **AC-12.** Given новый пароль короче 8 символов или поля «пароль» и «подтверждение» не совпадают, When отправляется запрос на смену, Then смена отклоняется с понятной ошибкой и пароль не меняется. *(FR-10, FR-11)*
- **AC-13.** Given выполняется любой шаг сброса, When инспектируются логи приложения, Then в них отсутствуют код/токен, новый пароль и `passwordHash`. *(FR-16)*
- **AC-14.** Given пользователь на странице логина, When он не помнит пароль, Then он видит ссылку «Забыли пароль?» и попадает в флоу сброса. *(FR-18, FR-19)*

## 8. Допущения и открытые вопросы

**Принятые допущения:**
- **A1.** Сброс идёт по 6-значному коду с TTL 15 мин (как verify-email), хранение по аналогии с `VERIFY#<email>/CODE` (например, `RESET#<email>`), с TTL DynamoDB. Выбор «код vs ссылка с токеном» и точная схема хранения — за архитектором; требования (криптостойкость, одноразовость, TTL, лимит попыток) остаются.
- **A2.** Политика нового пароля = текущей политике регистрации (минимум 8 символов). Усиление политики (спецсимволы, проверка по списку утёкших) — отдельное решение, в scope не входит.
- **A3.** Лимит попыток ввода кода по умолчанию 5; конкретные пороги rate limiting — за архитектором, важно лишь наличие и срабатывание (FR-6, FR-9).
- **A4.** Существующая генерация кодов использует `Math.random()` (register.ts). Для фичи аутентификации это недостаточно: код сброса генерируется криптостойким источником (FR-4). Замена генератора для verify-email — вне scope этой фичи (фиксируется как наблюдение для безопасника).
- **A5.** Сброс доступен только для аккаунтов с `emailVerified === true`; неподтверждённые аккаунты возвращаются в флоу верификации, а не сброса.
- **A6.** После успешного сброса допустимо сразу выдать новый JWT (авто-вход) ИЛИ отправить на экран логина — выбор за UX/архитектором; на требования не влияет (FR-15).
- **A7.** Партнёр в флоу сброса не участвует и не уведомляется; сброс — приватное действие одного пользователя, данные партнёра не затрагиваются.
- **A8.** Anti-timing «постоянное время ответа» реализуется так, чтобы не вредить UX легитимного пользователя (приемлемая фиксированная задержка / отсутствие раннего выхода) — детали за архитектором.

**Открытые вопросы:**
- [ ] **Механизм инвалидации сессий/JWT после смены пароля** (FR-14). JWT сейчас stateless (7d, без отзыва; `requireAuth` лишь верифицирует подпись). Нужен механизм отзыва — например, `passwordChangedAt`/`tokenVersion` в профиле, проверяемый при аутентификации, либо иной подход. Вопрос **к архитектору/безопаснику** (не к PO): спроектировать и оценить влияние на каждый аутентифицированный запрос. Брифом такое требование предусмотрено явно («инвалидация активных сессий/токенов после смены пароля — проработать с безопасником»).
- [ ] **Нужна ли защита от ботов сверх rate limiting** (капча) для публичного эндпоинта запроса сброса — на оценку архитектора/безопасника; по умолчанию для v1 достаточно rate limiting (капча — необязательна).
- [ ] **Информационное письмо «пароль изменён»** (FR-17) — оставить ли в v1. Рекомендация BA: включить (дёшево, заметно повышает безопасность/доверие). Решение за PO/архитектором; не блокирует.

> `[BLOCKER]` для PO отсутствуют — продуктовое направление однозначно (базовая ожидаемая функция), приватность и данные пользователей не расширяются, необратимых решений и заметных денег нет. Открытые вопросы носят инженерный характер и решаются на этапе DESIGN.
