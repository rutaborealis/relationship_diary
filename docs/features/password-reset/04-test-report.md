## Вердикт: PASS

**Повторный прогон после фикса `[BUG-1]`/SEC-01.** Закрыто 14 из 14 acceptance criteria. Ранее блокирующий **AC-8** (лимит попыток) перепроверен прицельно и теперь PASS: атомарный условный инкремент `attempts` стал гейтом ДО сравнения кода — ровно 5 попыток, после 5 неверных верный код больше не принимается (6-я отклонена + код удалён), пароль не меняется. Добавленный confirm rate-limit `RESET#<email>/RLC` (20/час) подтверждён на границе (20-я разрешена, 21-я блокирует даже верный код, гейт кода не достигается). Регрессия остальных 13 AC (anti-enumeration, одноразовость/TTL, resend-инвалидация, request rate-limit, смена пароля, инвалидация JWT через tokenVersion, политика пароля) — без регрессий. Остаточных блокирующих `[BUG]` нет.

- Локальный стек: DynamoDB Local (Docker, :8000), таблицы `DiaryMain-dev`/`DiaryPush-dev` созданы `setup-local-dynamo`. `:3000` занят (ssh/grafana) — Express-обёртку держим на :3001, но прицельная перепроверка AC-8/RLC и регрессия выполнены **прямым вызовом хендлеров** (`confirm-reset`, `request-reset`, `login`, `me`) против DynamoDB Local через CJS-хук (Node v23) — детерминированный контроль над счётчиком `attempts`, RLC и TTL. Resend замокан (`config.ses.localMode` → письма печатаются в консоль).
- Тест-харнесс: 32 ассерта (TEST A..G), все PASS. Записи `RESET#<email>/{CODE,RLC,RL}` и `tokenVersion` в `USER#<id>/PROFILE` инспектировались напрямую.

## Проверка приватности
Фича не читает и не пишет контент дневника; новых полей записи нет; партнёр в флоу не участвует и не уведомляется (NFR §6, A7). `request-reset`/`confirm-reset` трогают только `RESET#…`, `EMAIL#…`, `USER#…/PROFILE`.

## Регрессии (повторный прогон)
- Фикс затронул только `confirm-reset.ts` + порог `resetConfirmRateLimitPerHour` в config. Перепроверены все смежные AC прямым вызовом хендлеров:
  - AC-4/AC-5 (смена пароля): верный код → `200`, `passwordHash` обновлён, login старым → `401`, новым → `200 + token`. PASS.
  - AC-6 (одноразовость): повтор использованного кода → `400`. PASS.
  - AC-7 (TTL): `ttl` в прошлом → `400`, запись `CODE` удалена. PASS.
  - AC-11 (инвалидация JWT): старый токен `tv:1` до сброса → `/me` `200`; после сброса (`tokenVersion 1→2` через `if_not_exists+1`) → `401`. PASS — success-path гейта НЕ сломал бамп tokenVersion.
  - AC-12 (политика пароля): `newPassword` <8 → `400 "Password must be at least 8 characters"`; пропуск поля → `400 "Missing required fields"`; пароль не меняется. PASS.
  - AC-1/2/3 (anti-enumeration): verified/unverified/ghost → идентичный нейтральный `200`; `CODE` создан только для verified. PASS.
- Гейт «попытка тратится и на верный код»: success-path проходит тот же атомарный инкремент (TEST B: верный код на 5-й попытке инкрементит `attempts→5` перед сравнением и принимается). Фикс не ввёл регрессий в happy path.
- `tokenVersion`-логика и `requireAuth` (доп. `GetItem` профиля, ADR-0002) не менялись — работают как прежде.

## Дефекты
- `[BUG-1]` (medium, security) — **ИСПРАВЛЕНО и подтверждено.** Перепроверка: off-by-one устранён (ровно 5 сравнений), success-path теперь проходит атомарный гейт `attempts < :max` ДО сравнения, поэтому верный код после исчерпания лимита отклоняется (TEST A: 6-я попытка с верным кодом → `400` + код удалён). Дополнительно добавленный confirm rate-limit `RESET#<email>/RLC` (20/час) подтверждён (TEST C). Остаточных дефектов нет.

> Примечание про SEC-01 (HIGH, конкурентный обход лимита) — относится к параллельной security-перепроверке. С точки зрения QA: атомарность инкремента `ADD ... ConditionExpression` означает, что при N конкурентных запросах число успешных гейтов (и, следовательно, сравнений кода) ограничено `:max=5` на уровне DynamoDB. Динамический конкурентный нагрузочный PoC (40 параллельных) в этом прогоне не гонялся (см. «Не протестировано»); логически конкурентный обход закрыт тем же гейтом.

## Не протестировано
- **UI e2e через браузер** (Playwright): не прогонялся; флоу фронта проверен статически (build зелёный, ревью `ForgotPasswordPage`/`ResetPasswordPage`/`LoginPage`/`App.tsx`/`store`). Клиентские проверки FR-11 (совпадение паролей, длина ≥8, редирект при отсутствии `pendingEmail`) проверены по коду, не через рендер.
- **Реальная отправка Resend** — намеренно замокана (`localMode`), как требует ТЗ; письма (код сброса, «пароль изменён») проверены по содержимому в консоли, реальная доставка не проверялась.
- **Постоянство тайминга под нагрузкой / cold start** — замерены единичные значения (0.60–0.62с, неотличимы); статистический разброс под нагрузкой не профилировался.
- **`sam build` / деплой инфраструктуры** — не выполнялся (этап deploy); бэкенд тестировался через локальную Express-обёртку, не через реальный API Gateway.
- **FR-17 письмо «пароль изменён»** — отправляется (видно в моке после успешного сброса), best-effort-поведение при сбое отправки не инжектировалось.
- **Конкурентный нагрузочный PoC (N параллельных confirm на один код)** — в этом прогоне не запускался; защита от конкурентного обхода (SEC-01) проверена аналитически (атомарность `ADD attempts ConditionExpression attempts < :max`). Лид сообщает о прогоне 40-параллельных в self-check (03-build-notes §фикс, пункт C). Динамическая верификация конкуренции — за параллельной security-перепроверкой.
